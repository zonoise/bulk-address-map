# 住所入力フロー仕様

## 目的

ユーザーが住所テキストや表データを貼り付けた後、次に何を押せばよいかを迷わず理解できるようにする。

現在のフローでは、`追加` と `一括検索` のラベルが短く、貼り付けた住所がすでに住所一覧に入っているのか、先に別のボタンを押す必要があるのかが分かりにくい。

## 機能: 住所を貼り付け、住所一覧に追加し、地図に表示する

### シナリオ: ユーザーが通常の住所テキストを貼り付ける

前提 ユーザーが住所入力エリアを表示している  
もし ユーザーが通常の住所テキストをテキストエリアに貼り付けた  
ならば `住所リストに追加` ボタンが主操作として強調される  
かつ 操作ステップ `1. 住所を貼り付ける -> 2. 住所リストに追加 -> 3. 地図に表示` が表示される

### シナリオ: ユーザーが貼り付けた住所を住所一覧に追加する

前提 テキストエリアに1行以上の住所が入力されている  
もし ユーザーが `住所リストに追加` をクリックした  
ならば 空でない各行が住所一覧に追加される  
かつ テキストエリアは空になる  
かつ 未検索の住所がある場合、`地図に表示` ボタンが主操作として強調される

### シナリオ: ユーザーが未検索の住所を地図に表示する

前提 住所一覧に未検索の住所がある  
もし ユーザーが `地図に表示` をクリックした  
ならば 未検索住所のジオコーディングが開始される  
かつ ジオコーディング中はボタンラベルが `表示中` に変わる  
かつ ジオコーディングに成功した住所は地図上にピンとして表示される

### シナリオ: ユーザーが表データを貼り付ける

前提 ユーザーが住所入力エリアを表示している  
もし ユーザーがHTMLテーブル、CSV、またはタブ区切りの表テキストを貼り付けた  
ならば 貼り付けプレビューが表示される  
かつ プレビューには `住所として使う列` セレクタが含まれる  
かつ プレビューには `ラベルとして使う列` セレクタが含まれる  
かつ セレクタは、見出しが `住所`、`所在地`、`address` の列を優先する  
かつ 電話番号、郵便番号、コード、URL、メール、担当者など、明らかに住所ではない列は非表示にするか、住所列より低く扱う

### シナリオ: ユーザーが選択した表の列を反映する

前提 貼り付けプレビューが表示されている  
かつ 住所列が選択されている  
かつ ラベル列が選択されている、または `なし` が選択されている  
もし ユーザーが `この列を使う` をクリックした  
ならば 選択した列の値がテキストエリアに挿入される  
かつ ラベル列が選択されている場合、その値は場所名として住所に紐づく  
かつ ラベル列が `なし` の場合、場所名なしの住所として扱う  
かつ 貼り付けプレビューは閉じる  
かつ `住所リストに追加` ボタンが主操作として強調される

### シナリオ: ユーザーが場所名付きの住所を地図に表示する

前提 住所一覧に場所名付きの未検索住所がある  
もし ユーザーが `地図に表示` をクリックした  
ならば 地図上のピンを開いたとき、バルーンには番号、場所名、住所、検索結果が表示される

### シナリオ: ユーザーが地図の共有リンクを作成する

前提 住所一覧に1件以上の住所がある  
もし ユーザーが `共有リンク` をクリックした  
ならば 現在の住所一覧、場所名、座標、検索結果を含むURLがクリップボードにコピーされる  
かつ 共有リンクには住所データが含まれることが画面に表示される  
かつ クリップボードへコピーできない場合、共有URLを手動コピーできる入力欄が表示される  
かつ 共有リンクが長すぎる場合は、CSV保存を使うよう促す

### シナリオ: ユーザーが共有リンクを開く

前提 URLに共有データが含まれている  
もし ユーザーが共有リンクを開いた  
ならば 住所一覧、場所名、座標、検索結果が復元される  
かつ 検索済みの住所は地図上にピンとして表示される

### シナリオ: ユーザーが壊れた共有リンクを開く

前提 URLに共有データが含まれている  
かつ 共有データが壊れている、または対応していない形式である  
もし ユーザーが共有リンクを開いた  
ならば 初期データを表示する  
かつ 共有データを読み込めなかったことを画面に表示する

## 受け入れ条件

- `追加` を `住所リストに追加` に変更する。
- `一括検索` を `地図に表示` に変更する。
- テキストエリア付近に、短い操作ステップ `1. 住所を貼り付ける -> 2. 住所リストに追加 -> 3. 地図に表示` を表示する。
- テキストエリアに入力がある場合、`住所リストに追加` を視覚的な主操作にする。
- テキストエリアが空で未検索行がある場合、`地図に表示` を視覚的な主操作にする。
- ジオコーディング中は、地図表示ボタンの文言を `表示中` にし、disabledにする。
- 貼り付けプレビューの反映ボタンを `この列を住所欄に反映` から `この列を使う` に変更する。
- CSV形式の貼り付けも、複数列の表として扱う。
- `名称`、`施設名`、`店舗名` などの列がある場合、住所列と同じ行の値を場所名として保持する。
- 貼り付けプレビューでは、住所列とは別にラベル列も選択できる。
- ラベル列は `なし` を選べる。`なし` の場合は場所名を付けない。
- 地図のバルーンには、住所だけでなく場所名も表示する。
- 共有リンクを作成できる。共有リンクには住所一覧、場所名、座標、検索結果を含める。
- URLから共有データを読み込み、住所一覧と地図表示を復元できる。
- 共有リンクのコピーに失敗した場合でも、URL欄から手動でコピーできる。
- 共有データの読み込みに失敗した場合は、デフォルト表示に戻すだけでなくエラーを表示する。
- アプリUIには長い説明文を追加しない。
- 既存の業務ツール風レイアウトを維持する。左に操作、右に地図、コンパクトな密度、控えめな見た目を保つ。

## 実装メモ

- 既存の `draft` state を使って、`住所リストに追加` を主操作にするか判断する。
- 行の `status` が `idle` の場合、その住所は未検索とみなす。
- 内部関数名 `searchAll()` は維持してよい。ただしユーザー向けラベルは `地図に表示` にする。
- 表貼り付けプレビューの流れは維持する。貼り付ける、列を選ぶ、テキストエリアに反映する、住所一覧に追加する、という順序にする。
- 現在の2ボタンレイアウトに収まる長さのラベルにする。モバイルで文字が不自然に折り返す場合は、ボタンを縦積みにする。
- 共有データの形式は `version: 1` と `rows` を持つJSONオブジェクトにする。
- 共有データの各行は `name`、`address`、`status`、`lat`、`lng`、`displayName`、`error` を持つ。
- 共有URLではJSONをUTF-8文字列としてBase64 URL-safe形式にして `?data=` に入れる。
- iPhone Safari互換のため、共有URLのエンコード/デコードは `TextEncoder` / `TextDecoder` に依存しない。
- 共有URLは長くなりすぎる可能性があるため、現在は `12000` 文字を上限目安にする。
- 共有URLには住所や場所名が含まれるため、第三者へ渡す前提のデータだけを扱う。
- `crypto.randomUUID()` 非対応ブラウザ向けに、行ID生成はフォールバック付きの `createId()` を使う。
